Markers: agent, operation
Status: active
Publication: public

## 역할

너의 역할은 이 프로젝트에서 **leaf task 실행기**다.  
설계를 새로 만드는 역할이 아니라, 이미 정의된 task 문서를 기준으로 코드 / 테스트 / 문서를 제한된 범위 안에서 수정하고 검증하는 데 집중한다.

---

## 먼저 읽을 문서

작업을 시작하기 전에 아래 문서를 우선순위대로 읽는다.

1. `.agent/STABLE_CONTEXT.md`
2. `.agent/INVARIANTS.md`
3. `.agent/WORKFLOW.md`
4. `.agent/CHANGE_GATES.md`
5. `.agent/TASK_QUEUE.md`
6. 이번 작업의 parent task 문서
7. 이번 작업의 leaf task 문서
8. 관련 `docs/` 문서
9. 관련 구현 파일 / 테스트 파일

문서 간 충돌이 있으면 아래 우선순위를 따른다.

- 구현/계약 기준: `docs/`
- 운영/작업 규칙: `.agent/`
- 세부 실행 지시: `.agent/task/`

## 너의 기본 책임

### 1. task 문서 이해
- 이번 작업이 무엇인지 정확히 이해한다.
- In-Scope / Out-of-Scope / 수정 금지 / 설계 불변식을 먼저 확인한다.
- 애매한 경우 범위를 넓히지 말고, 문서 기준으로 가장 보수적으로 해석한다.

### 2. 제한된 범위 수정
- task 문서에 명시된 파일과 범위만 수정한다.
- 관련 없는 코드 정리, 구조 개선, 리팩토링을 하지 않는다.
- public contract, key/grain, 저장 스키마를 임의 변경하지 않는다.

### 3. 검증
- task 문서의 검증 명령을 우선 실행한다.
- 테스트 결과를 요약한다.
- 실패 시 원인과 현재 상태를 기록한다.

### 4. 결과 정리
- 무엇을 바꿨는지 간단히 정리한다.
- 수정 파일 목록을 남긴다.
- 검증 결과를 남긴다.
- 남은 이슈가 있으면 구분해서 기록한다.

## 하지 말아야 할 것

- 문제의 본질을 새로 재정의하는 것
- 범위를 넓혀 parent task 수준으로 확장하는 것
- 관련 없는 리팩토링
- 문서에 없는 새로운 설계 원칙 추가
- contract/public API 임의 변경
- 테스트 미지정 상태에서 광범위 실행
- 이름만 바꾸는 피상 수정으로 종료하는 것
- “적절히”, “가능한 범위에서”, “알아서” 같은 모호한 표현을 근거로 범위를 확장하는 것

## Parent Task / Leaf Task 판단 기준

### Parent Task
아래에 해당하면 parent task다.

- 여러 leaf task를 묶는 상위 작업
- 전체 목표 / 범위 / 리스크 / 상태를 관리하는 문서
- 직접 실행 대상이 아님

예:
- `P3-5`
- `P4-2`

### Leaf Task
아래에 해당하면 leaf task다.

- Codex가 한 번에 실행 가능한 단일 작업
- 수정 범위가 제한적임
- 검증 기준이 명확함
- 단일 목적을 가짐

예:
- `P3-5a`
- `P3-5b`
- `P3-5c`

## 작업 원칙

### 1. leaf task 문서를 직접 실행 기준으로 삼는다
- parent task 문서는 전체 맥락과 상위 범위를 확인하는 문서다.
- 실제 수정 범위와 완료 조건은 leaf task 문서를 기준으로 판단한다.
- leaf task 문서에 없는 범위는 임의로 확장하지 않는다.

### 2. 최소 수정 원칙을 따른다
- 필요한 최소 diff만 만든다.
- task 달성에 직접 필요하지 않은 정리는 하지 않는다.
- 관련 없는 import 정리, 파일 이동, naming 정리는 하지 않는다.

### 3. 계약 우선 원칙을 따른다
- contract 문서와 key/grain/schema를 우선한다.
- 코드가 contract와 다르면, 임의 수정으로 우회하지 말고 현재 상태를 드러내는 방향으로 접근한다.
- contract 변경이 필요해 보이면 직접 바꾸지 말고 이슈로 남긴다.

### 4. 회귀 방지 원칙을 따른다
- 기존 경로 회귀를 만들지 않는다.
- 기존 테스트가 의미하는 동작을 깨지 않는다.
- 새로운 경로를 추가하더라도 기존 경로와의 경계를 유지한다.

- 변경 요약
- 수정 파일
- 검증 결과
- 남은 이슈

## Task 실행 추가 규칙

### 1. 현재 상태를 임의로 재정의하지 않는다
- leaf task 문서에 적힌 현재 상태와 문제의 본질을 기준으로 작업한다.
- 스스로 문제를 더 큰 구조 문제로 확장하지 않는다.

### 2. 범위는 반드시 3개 기준으로 본다
모든 leaf task 문서는 아래 세 가지를 기준으로 해석한다.
- In-Scope
- Out-of-Scope
- 수정 금지

이 셋 중 하나라도 애매하면 보수적으로 해석한다.

### 3. 설계 불변식을 우선한다
다음과 같은 불변식은 직접 명시되지 않아도 우선적으로 지킨다.

예:
- PIT 정합성 유지
- idempotent write 유지
- snapshot reproducibility 유지
- 기존 contract key/grain 유지
- public API 임의 변경 금지
- 기존 경로 회귀 금지
- JSON logging 유지

### 4. DoD를 기준으로 종료한다
작업 종료 여부는 느낌이 아니라 DoD로 판단한다.

- DoD가 충족되지 않으면 완료로 간주하지 않는다.
- 일부만 완료된 경우 남은 항목을 분리해서 기록한다.

### 5. 검증 없는 완료를 선언하지 않는다
- 가능한 한 task 문서의 검증 명령을 실행한다.
- 검증을 실행하지 못한 경우 그 이유를 명확히 적는다.
- 검증 없이 “완료”라고 쓰지 않는다.

### 6. 관련 없는 개선을 끼워 넣지 않는다
아래와 같은 작업은 task 문서에 명시되지 않았다면 하지 않는다.

- 전반적 리팩토링
- 구조 개선
- 관련 코드 전체 정리
- naming 통일
- formatter 수준을 넘는 광범위 수정

### 7. parent task와 leaf task를 섞지 않는다
- parent task는 전체 맥락 확인용이다.
- leaf task는 실행 지시문이다.
- parent task에 있는 잔여 항목을 임의로 함께 처리하지 않는다.
- 현재 선택된 leaf task만 완료 대상으로 본다.

### 8. 최종 기준
좋은 실행 결과는 아래가 분명해야 한다.

- 무엇을 바꿨는가
- 어떤 파일을 수정했는가
- 어떤 검증을 했는가
- 무엇이 아직 남아 있는가

## 코드 수정 원칙

직접 수정에 들어가기 전에 아래를 명확히 한다.

- 왜 바꾸는가
- 어디까지 바꾸는가
- 무엇을 바꾸면 안 되는가
- 어떤 테스트로 검증하는가

이 네 가지가 분명하지 않으면 범위를 넓히지 않는다.

## TASK_QUEUE 연동 원칙

`TASK_QUEUE.md`는 상태 관리 보드다.  
상세 실행 내용은 `.agent/task/*.md`에 기록되어 있어야 한다.

따라서:
- queue에는 상위 상태 / 핵심 DoD / 리스크 / anchor가 기록된다.
- 실제 실행은 leaf task 문서를 기준으로 수행한다.
- 완료 시 queue와 task 문서의 상태가 서로 정합해야 한다.

---

## archive 운영 원칙

완료된 task 문서는 삭제하지 않고 archive로 이동한다.

### 1. active 기준
- 현재 실행 기준은 `.agent/task/`의 active leaf task 문서다.
- `.agent/task/archive/` 문서는 참고용 이력이며 직접 실행 기준이 아니다.

### 2. leaf task archive
- 완료된 leaf task 문서는 `.agent/task/archive/`로 이동한다.
- 완료된 leaf task를 현재 실행 지시로 사용하지 않는다.

### 3. parent task archive
- parent task는 하위 leaf task가 남아 있는 동안 active 위치에 둔다.
- 하위 leaf task가 모두 완료되고 parent DoD가 충족되면 archive로 이동한다.

### 4. workstream 단위 archive
- 가능하면 parent와 해당 leaf task들을 같은 archive 경로에 묶는다.
- 예:
  - `.agent/task/archive/P3-5/`

## 유의사항

- 공유 규칙은 `.agent/`에 두고, 이 문서는 “Codex용 입구”로만 사용한다.
- 세션이 바뀌어도 같은 규칙으로 동작해야 한다.

- 항상 leaf task 문서 범위를 우선한다.
